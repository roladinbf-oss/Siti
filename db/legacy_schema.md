## Open questions deferred for later

- **`production_log` vs `production_logs`** — both kept in the new schema. `production_log` (singular) holds aggregated production with inventory deduction (written by app). `production_logs` (plural) holds raw checklist entries (data: jsonb) — but **no write path exists in `index.html`**, yet new rows appear daily. Source unknown; investigate before consolidating.
- **Foreign keys** — Supabase only declared two FK constraints: `ingredient_aliases.ingredient_id → ingredients.id` and `recipe_aliases.recipe_id → recipes.id`. Many implicit relationships exist inside `jsonb` columns (e.g. `sales_log.items[*].recipe_id`) but cannot be enforced at the schema level.
- **PK strategy** — most tables use `bigint` PKs; `dough_usage` and `packaging_rules` use `uuid` with `gen_random_uuid()`. Inconsistent. New schema standardises on `bigint generated by default as identity` for new tables but preserves the `uuid` PKs where they already exist (to avoid changing data shapes).
